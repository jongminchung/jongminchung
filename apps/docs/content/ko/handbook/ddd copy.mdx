> 비즈니스 규칙을 gRPC 핸들러와 DB 코드의 if 문에 흩어놓지 말고, 도메인 객체가 자신의 상태와 상태 변경 규칙을 책임지게 하자.

글은 이를 세 가지 규칙으로 설명합니다.

1. 비즈니스 행동을 코드에 그대로 표현
2. 메모리에 존재하는 객체는 항상 의도된 상태여야 함
3. 도메인 모델을 데이터베이스로부터 분리

## 예제 도메인

트레이너의 특정 시간대를 나타내는 Hour가 있다고 하겠습니다.

가능한 상태는 다음과 같습니다.

| **현재 상태** | **허용되는 행동** | **결과**  |
| ------------- | ----------------- | --------- |
| 예약 가능     | 운동 예약         | 예약됨    |
| 예약됨        | 예약 취소         | 예약 가능 |
| 예약 불가     | 운동 예약         | 오류      |
| 예약 가능     | 예약 불가 처리    | 예약 불가 |

이 규칙이 이 서비스의 핵심 비즈니스 로직입니다.

### 잘못된 예시: 객체를 단순 데이터로 취급

```go
type Hour struct {
    Time time.Time
    Available bool
    HasTrainingScheduled bool
}
```

핸들러에서 직접 상태를 변경합니다.

```go
func (h *Handler) UpdateHour(
    ctx context.Context,
    req UpdateHourRequest,
) error {
    hour, err := h.db.FindHour(ctx, req.Time)
    if err != nil {
        return err
    }

    if req.HasTrainingScheduled && !hour.Available {
        return errors.New("예약할 수 없는 시간입니다")
    }

    if req.Available && req.HasTrainingScheduled {
        return errors.New("예약된 시간을 예약 가능으로 만들 수 없습니다")
    }

    hour.Available = req.Available
    hour.HasTrainingScheduled = req.HasTrainingScheduled

    return h.db.SaveHour(ctx, hour)
}
```

### 문제 1: 잘못된 상태를 만들 수 있다.

두 개의 bool을 사용하면 네 가지 조합이 만들어집니다.

당장은 단순해 보이지만 여려 문제가 있습니다.

| **Available** | **HasTrainingScheduled** | **의미**                   |
| ------------- | ------------------------ | -------------------------- |
| true          | false                    | 예약 가능                  |
| false         | true                     | 예약 됨                    |
| false         | false                    | 예약 불가 또는 의미 불명확 |
| true          | true                     | 모순된 상태                |

컴파일러는 `true, true`를 막아주지 않습니다.

```go
hour.Available = true
hour.HasTrainingScheduled = true
```

객체가 자신의 규칙을 보호하지 못하고 있습니다.

### 문제 2: 코드가 비즈니스 언어를 말하지 않는다.

비즈니스 담당자는 보통 이렇게 말합니다.

> “오후 1시 운동을 예약합니다.”

하지만 코드는 이렇게 말합니다.

> “Available을 false로 설정하고 HasTrainingScheduled를 true로 설정합니다.”

이 때문에 상태를 변경하는 기술적인 절차를 읽고 비즈니스 행동을 머릿속에서 다시 해석해야 합니다.

### 좋은 예시 1: 상태를 하나의 개념으로 모델링

두 개의 `bool` 대신 도메인에서 사용하는 상태를 명시적으로 표현합니다.

```go
type Availability uint8

const (
    AvailabilityUnknown Availability = iota
    Available
    NotAvailable
    TrainingScheduled
)
```

이제 Available && TrainingScheduled 같은 모순된 조합 자체가 사라집니다.

```go
type Hour struct {
    at           time.Time
    availability Availability
}
```

필드를 비공개로 만들었기 때문에 외부 패키지는 다음과 같은 코드를 작성할 수 없습니다.

```go
hour.availability = TrainingScheduled // 컴파일 오류
```

### 좋은 예시 2: 생성할 때부터 유효하게 만든다.

```go
var ErrNotFullHour = errors.New("정각만 등록할 수 있습니다")

func NewAvailableHour(at time.Time) (*Hour, error) {
    if at.Minute() != 0 || at.Second() != 0 {
        return nil, ErrNotFullHour
    }

    return &Hour{
        at:           at,
        availability: Available,
    }, nil
}
```

다음처럼 잘못된 `Hour`를 먼저 만들고 나중에 검증하지 않습니다.

```go
// 잘못된 방식
hour := Hour{at: time.Date(2026, 7, 20, 13, 37, 0, 0, time.UTC)}

if err := validate(hour); err != nil {
    // 이미 잘못된 객체가 메모리에 존재한다.
}
```

좋은 방식에서는 생성에 성공했다면 최소한 생성 규칙을 통과한 객체라고 믿을 수 있습니다.

```go
hour, err := NewAvailableHour(at)
if err != nil {
    return err
}

// 이 시점의 hour는 유효하다.
```

이것이 글에서 말하는 “항상 유효한 상태를 메모리에 유지한다”는 뜻입니다.

### 좋은 예시 3: setter 대신 비즈니스 행동을 제공한다

잘못된 설계는 범용 setter를 제공합니다.

```go
func (h *Hour) SetAvailability(v Availability) {
    h.availability = v
}
```

그러면 호출자가 비즈니스 규칙을 알아야 합니다.

```go
if hour.Availability() == Available {
    hour.SetAvailability(TrainingScheduled)
}
```

좋은 설계에서는 객체가 규칙을 책임집니다.

```go
var (
    ErrHourNotAvailable   = errors.New("예약 가능한 시간이 아닙니다")
    ErrNoTrainingScheduled = errors.New("예약된 운동이 없습니다")
)

func (h *Hour) ScheduleTraining() error {
    if h.availability != Available {
        return ErrHourNotAvailable
    }

    h.availability = TrainingScheduled
    return nil
}

func (h *Hour) CancelTraining() error {
    if h.availability != TrainingScheduled {
        return ErrNoTrainingScheduled
    }

    h.availability = Available
    return nil
}
```

호출하는 쪽도 비즈니스 문장처럼 읽힙니다.

```go
if err := hour.ScheduleTraining(); err != nil {
    return err
}
```

중요한 차이는 단순히 코드가 짧아졌다는 것이 아닙니다.

> Hour의 상태를 바꾸는 모든 경로가 Hour의 규칙을 통과한다.

### 좋은 예시 4: 범용 Update API를 행동별로 API로 나눈다

잘못된 API는 클라이언트가 내부 상태 조합을 결정하게 합니다.

```protobuf
rpc UpdateHour(UpdateHourRequest) returns (EmptyResponse);

message UpdateHourRequest {
  Timestamp time = 1;
  bool available = 2;
  bool has_training_scheduled = 3;
}
```

클라이언트가 다음을 알아야 합니다.

- 예약하려면 어떤 bool 조합을 보내야 하는가?
- 취소하려면 어떤 조합인가?
- `true, true`를 보내면 어떻게 되는가?

좋은 API는 의도를 직접 표현합니다.

```protobuf
rpc ScheduleTraining(UpdateHourRequest) returns (EmptyResponse);
rpc CancelTraining(UpdateHourRequest) returns (EmptyResponse);
rpc MakeHourAvailable(UpdateHourRequest) returns (EmptyResponse);

message UpdateHourRequest {
  Timestamp time = 1;
}
```

API만 보더라도 시스템에서 가능한 행동을 알 수 있습니다. 원문도 UpdateHour를 위 세 가지 행동으로 분리합니다.

---

하위 계층은 범용적이고 조합 가능한 메커니즘을 제공합니다.

그 위의 도메인 계층은 정책을 표현합니다.

```text
클라이언트의 의도
    ScheduleTraining
           ↓
도메인의 정책
    Hour.ScheduleTraining
           ↓
범용 저장 메커니즘
    Repository.Update
           ↓
DB의 원시 연산
    Transaction / Put
```

---

애플리케이션 로직과 데이터베이스 로직이 결합되면 애플리케이션이 복잡해집니다.

```go
func (s *Server) ScheduleTraining(
    ctx context.Context,
    at time.Time,
) error {
    // DB 조회
    // 예약 가능 여부 검사
    // 상태 변경
    // DB 저장
    return nil
}
```

문제는 서비스가 성장하면서 이 함수가 다음을 전부 담당하게 된다는 것입니다.

- SQL
- 데이터베이스 스키마
- 도메인 객체 변환
- 비즈니스 규칙
- 트랜잭션
- 동시성 제어
- 오류 응답 변환

이 문제를 Repository로 분리합니다.

하지만 Repository의 목적은 단순히 SQL을 다른 파일로 옮기는 것이 아닙니다.

> 애플리케이션 "어떻게 저장하는가"가 아니라 "어떤 도메인 객체를 가져와 무엇을 하는가"에만 집중하게
> 만드는 것이 핵심이다.

### 잘못된 예시 1: 핸들러가 DB와 비즈니스 규칙을 모두 담당한다

```go
func (s *Server) ScheduleTraining(
    ctx context.Context,
    at time.Time,
) error {
    var availability string

    err := s.db.GetContext(
        ctx,
        &availability,
        `SELECT availability
           FROM hours
          WHERE hour = ?`,
        at,
    )
    if err != nil {
        return fmt.Errorf("시간 조회 실패: %w", err)
    }

    if availability != "available" {
        return ErrHourNotAvailable
    }

    _, err = s.db.ExecContext(
        ctx,
        `UPDATE hours
            SET availability = 'training_scheduled'
          WHERE hour = ?`,
        at,
    )
    if err != nil {
        return fmt.Errorf("시간 저장 실패: %w", err)
    }

    return nil
}
```

핸들러가 너무 많은 것을 안다

- 테이블 이름은 `hours`다.
- 컬럼 이름은 `availability`다.
- 예약 가능 상태는 `"available"` 문자열이다.
- 예약 완료 상태는 `"training_scheduled"`다.
- MySQL의 `SELECT`, `UPDATE`를 사용한다.
- 데이터베이스 문자열을 비즈니스 상태로 해석해야 한다.

비즈니스 담당자가 궁금한 것은 이것입니다.

> 언제 운동을 예약할 수 있나요?

그런데 답을 찾으려면 SQL과 핸들러 코드를 함께 읽어야 합니다.

SQL과 핸들러 코드가 같이 존재하면 데스트에도 DB가 필요해집니다.

```go
func TestScheduleTraining(t *testing.T) {
    db := startMySQL(t)
    seedHour(t, db, NotAvailable)

    server := NewServer(db)

    err := server.ScheduleTraining(ctx, at)

    require.ErrorIs(t, err, ErrHourNotAvailable)
}
```

비즈니스 규칙 테스트가 통합 테스트가 되어버립니다.

### 잘못된 예시 2: 이름만 Repository인 DB래퍼

다음처럼 SQL을 인터페이스 뒤에 숨기면 Repository가 된 것처럼 보입니다.

```go
type Repository interface {
    FindOne(
        ctx context.Context,
        table string,
        where map[string]any,
    ) (map[string]any, error)

    Update(
        ctx context.Context,
        table string,
        values map[string]any,
        where map[string]any,
    ) error
}
```

사용하는 쪽은 다음과 같습니다.

```go
row, err := repo.FindOne(
    ctx,
    "hours",
    map[string]any{"hour": at},
)

if row["availability"] != "available" {
    return ErrHourNotAvailable
}

return repo.Update(
    ctx,
    "hours",
    map[string]any{
        "availability": "training_scheduled",
    },
    map[string]any{
        "hour": at,
    },
)
```

구현체를 인터페이스로 감쌌지만 애플리케이션을 여전히 DB를 생각하고 있습니다.

- 테이블
- 컬럼
- 조건절
- 문자열 상태
- `map[string][any]`

이것은 Repository라기보다 범용 DB 클라이언트에 가깝습니다.

좋은 Repository 인터페이스는 저장 기술의 언어가 아니라 도메인의 언어를 사용해야 합니다.

### 좋은 예시 1: 도메인 객체를 기준으로 인터페이스를 정의한다

```go
type Repository interface {
    GetOrCreateHour(
        ctx context.Context,
        at time.Time,
    ) (*Hour, error)

    UpdateHour(
        ctx context.Context,
        at time.Time,
        updateFn func(*Hour) (*Hour, error),
    ) error
}
```

애플리케이션이 볼 수 있는 것은 Hour뿐입니다.

```go
h, err := repository.GetOrCreateHour(ctx, at)
if err != nil {
    return err
}

return h.IsAvailable()
```

이 코드를 봐서는 MySQL인지 Mongo인지 알 수 없습니다. 바로 그것이 의도입니다.

- 조직이 외부 변화를 감지하고, 판단하고, 소프트웨어를 통해 대응하는 전체 피드백 루프를 단축하는 것
- 다음 문제를 해결하기 위해 등장한 역량과 실천의 집합으로 설명
  - 안전하고
  - 복원력이 있으며
  - 빠르게 변화하고
  - 대규모로 운영되는 분산 시스템을 어떻게 만들 것인가?
- 측정이 필요함
  - 경영진이 DevOps 전환 진척을 실무자보다 높게 평가하는 경향
  - 평가가 달라지므로 공통된 측정 체계가 필요하다는 것
  - 어떤 역량이 성과를 예측하는지 연구로 선별해야 함
  - 고성과 집단은 배포 빈도가 높으며, 변경 실패율이 낮으며, 실패가 발생해도 서비스 복구 시간이 빠름

- 무엇을 성과로 정의하느냐가 조직의 행동을 어떻게 바꿀 수 있을까?
  - 기존 생산성이 실패하는 이유는 두 가지로 압축됨
    - 결과가 아니라 산출량을 측정함
    - 전체 시스템이 아니라 개인이나 부서를 측정함
  - 코드량은 비즈니스 가치와 단순 비례하지 않음
    - 많이 작성하면 불필요한 큰 코드와 유지보수 비용이 생김
    - 적게 작성하도록 보상해도 이해하기 어려운 과도한 압축이 생김
    - 코드를 삭제하거나 업무 절차를 바꿔 해결하는 더 좋은 방법은 평가하지 못함
